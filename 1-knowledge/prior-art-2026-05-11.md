## agents.yml への追加提案 (Phase 1 内)

```yaml
# 新規追加候補
structural-analyzer:
  name: 構造解析
  summary: Tree-sitter で関数依存・共有 util を検出 (embedding 補完)
  llm:
    primary: claude
  skills:
    - tree-sitter-query
    - graph-cypher
  knowledge_scope:
    - ~/.kb/repos/
  triggers:
    keywords: [依存, 影響, blast, util, 関数, 呼び出し]
    after: [coder]    # 実装後に必ず影響解析
  output_format:
    - 変更影響 repo/file 一覧
    - 依存関数グラフ (mermaid)
    - blast radius スコア

impact-analyst:
  name: 影響度評価
  summary: PR 前 blast radius 解析、reviewer 補助
  llm:
    primary: claude
    fallback: gemini
  skills:
    - dep-graph-query
  knowledge_scope:
    - ~/.kb/repos/
    - ~/.kb/prs/
  triggers:
    before: [reviewer]    # reviewer 前に必須
  output_format:
    - 影響範囲: high/medium/low
    - 関連 PR/Issue
    - 推奨テスト対象
```

→ **7 役 → 9 役に拡張**。Kuuki Design 18-agent 理想への到達率を 70% → 78% に押し上げ。

## Gemini 設計検証 (2026-05-11) — 新発見 3 件

Gemini は本 Phase 1 アプローチを「システムアーキ観点で**極めて理にかなった洗練された設計戦略**」と評価。
ハード追加前に「司令塔 + ハーネス」を組む = ソフトウェア工学の王道、と支持。

### G1: GitHub Issue を「外部の脳 (共有メモリ)」として運用 ★★★★★

> エージェントが直接すべてを記憶するのではなく、GitHub Issue や専用 Markdown ファイルを「外部の脳」として使います。エージェント A が調査結果を Issue に書き込み (R66 徹底)、エージェント B はそのIssue だけを読み込んで次の作業 (実装 / レビュー) を行う。
> → ローカルマシン負荷は「テキストの受け渡し」のみになり劇的に軽量化。

**本ハーネスへの統合**:
- `protocol.md` に「Issue-as-shared-memory」節を追加
- agents.yml: researcher → architect → coder の handoff を Issue 経由に統一
- historian 役: 最終結果を Issue/PR に書込んで「外部脳」更新

### G2: Skills 実行の安全防波堤 ★★★★★

> 最も危険なのが「意図しない本番環境の破壊」。
> クラウド LLM が直接コードを書換えるのではなく、LLM には GitHub CLI のコマンドやスクリプト実行計画だけを出力させ、ローカル PC 側スクリプトが受取り、人間 (Captain) の承認トリガー (pull-based) を経てから安全に実行する「防波堤」を組み込む。

**本ハーネスへの統合**:
- `protocol.md` に「Safety Breakwater」節を追加
- route.sh / route.ps1: dry-run モードを既定、実行は `--execute` 明示
- R10 Batched Authorization を「コマンド一括承認」フォーマットに拡張

### G3: モデル振り分けロジック ★★★★

> どのタスクに高価なモデル (Opus 等) を使い、どのタスクを安価なモデル (Haiku 等) で済ませるかの「振り分けロジック」を Dify 上で確立する。

**本ハーネスへの統合**:
- `agents.yml` の `llm.primary` を **タスク重量別**に細分化
- 重: claude-opus / 中: claude-sonnet / 軽: claude-haiku, gemini-flash
- routing.yml の rule 毎にモデル指定

### Phase 1 完了の Gemini 提案目標

1. **「1 つの Issue 起点の完全自動リレー」完成**
   Issue 「○○ 機能追加」 → ①要件定義 AI → ②実装 AI → ③レビュー AI → ④結果 Issue 返却 を API 経由で完遂
2. **トークン消費最適化**: モデル振り分け確立
3. **API 向き先を localhost に変えるだけで Phase 2 移行可**な疎結合設計

→ 本目標を Phase 1 KPI として `protocol.md` に追記する。

## ChatGPT 技術詳細 (2026-05-11) — retrieval policy per role

ChatGPT は本方針を「**model size より harness quality で再現する実験**」として支持。
低 GPU 環境では「LLM を鍛えるより、役割定義・KB 検索・Skills・Issue 駆動・評価ログを鍛える方が費用対効果が高い」。

### C1: 全 Agent に同じ KB 検索を食わせない ★★★★★

> Agent の違いは prompt だけでは弱い。**検索対象・検索クエリ・評価軸・出力フォーマットまで分ける**と、かなり専門 Agent っぽくなる。

**新提案ファイル**: `~/.kb/config/agent_profiles.yaml` (retrieval policy 切り出し)

```yaml
architect:
  top_k: 12
  collections: [issues, decisions, r_rules]
  output: [decision, tradeoff, risk, next_action]
reviewer:
  top_k: 10
  collections: [past_bugs, r_rules, similar_fixes]
  output: [blocking, non_blocking, test_needed, rule_candidate]
implementer:
  top_k: 8
  collections: [code_patterns, current_issue, similar_impl]
  output: [patch_plan, files, commands, verification]
```

→ `search_kb(query, role)` で role 別 retrieval policy 切替。
→ feedback.sqlite3 に `role` カラム追加 (`ALTER TABLE feedback ADD COLUMN role TEXT DEFAULT 'default';`)。

### C2: 5 役 → 3 役で十分 (cognitive load 削減) ★★★★

> 1 人開発では Agent 管理自体が新しい仕事になる。最初は **Architect / Reviewer / Implementer の 3 役** で十分。

**本ハーネスへの反映**: 7 役は維持 (機能的に必要) だが、`agent_profiles.yaml` の MVP は ChatGPT 推奨 3 役 + Knowledge Curator (historian) = 4 役で開始。残りは Phase 2 で fine-tune。
